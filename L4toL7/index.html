FortiGate四层转七层后问题总结

背景​

应业务需要，FortiGate防火墙需要从常用的四层模式转换成七层模式。FortiGate防火墙从7.0.12升级到7.2.10，升级完成后转换到七层模式，重新添加策略。

问题1：特定端口业务异常​

现象​

FortiGate防火墙下一个8008端口的业务出现问题，通过防火墙抓包发现有流量转发，服务器端抓包，只有三次握手的包。并且防火墙日志中查看不到8008端口的日志（由于只有三次握手的包，并没有建立会话，所以没有生成日志）。

原因分析​

通过查资料，和厂商确认，是防火墙的一些特殊端口覆盖用户端口导致。TCP port 8008、8010、8015和8020这些端口是防火墙使用的端口，四层模式下只有在做VIP的端口冲突下，会被防火墙这些端口覆盖掉导致业务不通。七层模式下直接有这些端口的业务，也会被覆盖掉。

解决方案​

关闭这些防火墙端口对用户端口的覆盖。

操作步骤如下：

config webfilter fortiguard
set close-ports enable
end

问题2：DHCP中继阻断问题​

现象​

FortiGate下出现DHCP中继阻断问题，在转换模式前，DHCP中继是正常的，转换后发现终端获取不到IP，在防火墙中查看DHCP中继与DHCP server的会话被阻断，查看会话，发现同源IP的会话有多个，并且会话一直没有中断。

在有会话的情况下，新建策略也无法被命中。判断应该是防火墙转换模式后业务断开，但是会话没有断开，并且会话一直保持触发状态，成为一个无效的会话，导致新策略无法被命中，默认就阻断了。

解决方案​

清除掉这些无效会话，让新会话重新命中放行策略即可。

操作步骤如下：

  1. 设置会话过滤条件
# diagnose sys session filter src 192.168.x.x
  1. 查看过滤条件设置是否完成
# diagnose sys session filter 
session filter:
        vd: any
        sintf: any
        dintf: any
        proto: any
        proto-state: any
        source ip: 192.168.x.x-192.168.x.x
        NAT'd source ip: any
        dest ip: any
        source port: any
        NAT'd source port: any
        dest port: any
        policy id: any
        expire: any
        duration: any
        state1: any
        state2: any
  1. 查看条件过滤完成的会话
# diagnose sys session list
  1. 清除过滤出来的会话
# diagnose sys session clear
  1. 操作完成后，清除过滤条件
# diagnose sys session filter clear

重要提示​

注意：查看会话列表和清除会话时，最好是配合上过滤条件使用，不然很可能会导出所有的会话，导致设备内存使用过高或者清除所有的会话。

会话查看和删除的详细说明可参考FortiGate官方文档。
